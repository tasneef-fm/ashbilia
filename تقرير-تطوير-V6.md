# تقرير تطوير وردة أشبيليا V6

## التطويرات المنفذة

1. **مركز جودة البيانات**: صفحة جديدة تكشف العملاء المكررين، المنتجات بلا أسعار أو تكلفة، اختلاف أرصدة المخزون، عدم تطابق مالية الطلبات، الحجوزات المؤكدة بلا عربون مكتمل، الدفعات غير المرتبطة، أوامر العمل بلا توثيق، الحجوزات المنتهية للمعدات، أوامر الشراء المتأخرة، والحسابات غير المربوطة بموظفين.
2. **منع التكرار**: توسعة مفاتيح `idempotency_key` للحجوزات والمدفوعات وحركات المخزون وأوامر الشراء وأوامر العمل والتوصيل.
3. **انضباط الحالات**: جدول موحد لانتقالات الحالات، تحقق في قاعدة البيانات، وعرض الحالات المسموحة فقط في الواجهة.
4. **المصدر المالي الموحد**: دالة `recalculate_order_financials` ومزامنة تلقائية بعد تعديل بنود الطلب أو الدفعات.
5. **دقة العملاء**: توحيد أرقام الجوال السعودية ومنع إنشاء عميل جديد بنفس الرقم بعد التطبيع.
6. **سرعة القوائم**: ترقيم من السيرفر بواقع 40 سجلًا للصفحة وبحث مؤجل في المنتجات والطلبات والحجوزات والعملاء.
7. **حفظ المسودات**: استعادة تلقائية للنماذج غير المكتملة، مع استثناء كلمات المرور والملفات.
8. **مراقبة الأخطاء**: تجميع أخطاء الواجهة حسب البصمة بدل إنشاء مئات السجلات المتكررة.
9. **فهارس الأداء**: فهارس للعملاء والطلبات والحجوزات والمدفوعات وأوامر العمل والمشتريات والإشعارات.
10. **سجل رقابة**: كل إصلاح أو تجاهل موثق في مركز الجودة يسجل في `audit_logs`.

## اختبارات محلية منفذة

- فحص نحوي لجميع ملفات JavaScript عبر `node --check`.
- فحص وجود روابط API وRPC الجديدة داخل الواجهة والخدمة.
- فحص بنية ZIP والملفات الأساسية.
- فحص توازن محددات SQL الأساسية (`$$` والمعاملات) آليًا.

## ما يحتاج فحصًا داخل مشروع Supabase

لا يمكن تشغيل الترحيل على قاعدة المستخدم من هذه البيئة. يجب نجاح `verify-data-quality-v6.sql` ثم تجربة عملية طلب وحجز ومخزون على بيانات اختبار قبل الاعتماد التجاري.
